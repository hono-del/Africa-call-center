# 00. 現行モック分析

> 対象: https://cmc-ai-callcenter.vercel.app （日本のトヨタ系ディーラー想定モック）
> 本ドキュメントは、既存モックを外部調査した結果に基づく分析です。ソースコード自体は未参照のため、画面遷移・表示内容から読み取れる範囲でまとめています。

## 1. 現在のモックの目的

自動車ディーラー／メーカーのコールセンター業務を想定し、生成AIを活用した**オペレーター支援・分析ダッシュボード**のイメージを伝えるモックとして作られている。実データではなく、車種（プリウス、カムリ、RAV4、bZ4X等）や問い合わせカテゴリ（メンテナンス、EV充電、安全機能等）を模したサンプルデータで動作する。

想定ユースケース:
- オペレーターが通話中にAIからリアルタイム支援を受ける
- 管理者・SVがKPI（対応件数・解決率・満足度）を把握する
- 過去の問い合わせ傾向からナレッジギャップを発見する
- ギャップに対する改善アクション（動画制作・FAQ拡充等）をAIが提案する
- 解説動画に多言語字幕をAIで自動生成する

## 2. 主要画面

| # | 画面名 | URL | 概要 |
|---|--------|-----|------|
| 1 | ダッシュボード | `/dashboard` | KPIサマリー（対応件数・解決率・AI支援率・満足度）、カテゴリ別件数、解決ステータス内訳、日別推移、満足度分布、通話履歴一覧 |
| 2 | コール対応LIVE | `/call` | 車種選択、会話ログ表示、顧客発言入力、「AI分析」ボタンによるAI回答・参照リソース表示 |
| 3 | 質問分析 | `/analysis` | カテゴリ別パフォーマンス表、ナレッジギャップ一覧（優先度付き）、レーダーチャート、車種×カテゴリのヒートマップ、要フォローアップ一覧 |
| 4 | AI改善提案 | `/suggestions` | 改善アクション（動画制作/FAQ拡充/資料作成/トレーニング等）の一覧、優先度・カテゴリ・影響件数、ステータス管理 |
| 5 | 動画キャプション | `/captions` | 動画選択、チャプター表示、多言語（日/英/中/韓）キャプション生成 |

共通UI: ヘッダー（ロゴ、言語切替、メニュー、ログインユーザー、本日の対応件数）。

## 3. 流用できるUI/ロジック

- **画面構成の骨格**: ダッシュボード（KPI把握）→分析（傾向・ギャップ発見）→改善提案（アクション化）という循環構造は、業種・地域を問わず有効
- **KPIの考え方**: 対応件数、解決率、AI活用率、満足度という評価軸そのもの
- **優先度・ステータスの3段階設計**: 高/中/低、未対応/対応中/完了、解決済み/エスカレ/保留中という運用しやすい粒度
- **ナレッジギャップ検出→改善提案という一連のワークフロー**: 「分析結果から自動的に改善アクションを導く」という構造は、アフリカPoCでも中核となる価値提案と一致
- **可視化パターン**: レーダーチャート、ヒートマップ、分布図といった見せ方
- **コンポーネント設計方針**: 共通レイアウト、カード/バッジ/優先度ラベル等のUIパーツ分離

## 4. 作り替えるべき点

| 観点 | 現状 | 課題 |
|---|---|---|
| 中核画面の粒度 | `/call`は「会話ログ＋単一AI分析ボタン」 | 検索キーワード・複数回答候補・確認質問・次アクション・解決記録という要素が分離されておらず、今回伝えたい業務フローを表現できない |
| 車種・ブランドマスタ | プリウス、カムリ、RAV4、bZ4X等、日本・北米寄り | アフリカ市場の実態（トヨタ ハイラックス中心、スズキ急成長等）と乖離 |
| 問い合わせカテゴリ | メンテナンス、EV充電、安全機能等 | アフリカ特有の課題（部品供給・納期、悪路耐久性、並行輸入車対応等）が反映されていない |
| 対応言語 | 日本語/英語/中国語/韓国語 | 英語・フランス語・日本語の3言語に絞り込みが必要 |
| 対応チャネル | 音声通話のみ（想定） | テキストチャネル（WhatsApp/SMS等）を追加する必要あり |
| 動画キャプション機能 | 独立画面として存在 | 今回のPhase 1のスコープ外。除外する |
| ログデータモデル | 検索キーワード・回答候補・確認質問等のフィールドが存在しない可能性が高い | 新規に設計・追加が必要 |
| 解決結果の記録 | ステータス概念はあるが、担当者が明示的に入力する導線が不明 | 通話終了時の必須入力として再設計が必要 |
| ユーザー・権限 | オペレーター名がハードコードされている様子 | 将来のロールベース（担当者/管理者/販売店スタッフ/エンドユーザー）を見据えた設計に変更 |

## 5. まとめ

既存モックの**画面構成の骨格とKPI・分析・改善提案という循環構造は7割程度流用可能**。一方、**今回最も伝えたい中核体験（検索→複数回答候補→確認質問→次アクション→解決記録）を表現する`/call`画面は、実質的に作り直しに近い再設計が必要**。動画キャプション機能はスコープ外として除外する。
