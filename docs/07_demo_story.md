# 07. デモストーリー

## 本ストーリーの目的

1. **顧客がPoC実施のメリットを体感できること**
   - 問い合わせ対応時に情報を探す時間が削減できる
   - 解決率を上げるための「問い合わせ対応の勝ちパターン」が蓄積される
   - 回答できなかった・解決できなかったログを蓄積・分析し、ナレッジギャップの特定と解消につなげられる
2. **顧客とイメージを握り、このモックが要件定義のベースとなること**
   - 各シーンに「要件ディスカッションポイント」を用意し、デモを見ながらその場で要件の当たりをつけられる構成とする

デモは実装済みモックの実画面（`/assist`, `/dashboard`）と、組み込み済みデモケース（case_001, case_005, case_008）をそのまま使って進行できる。

---

## 登場人物（ペルソナ）

| 人物 | 役割 |
|---|---|
| **Amina**（アミナ） | 入社3か月目のコールセンター担当者。製品知識がまだ浅く、対応のたびにマニュアルやベテランへの確認で時間がかかっている |
| **Joseph**（ジョセフ） | コールセンター管理者（SV）。担当者ごとの対応品質のばらつきと、「何が解決できていないのか」が見えないことに課題を感じている |
| 顧客A | ケニアの顧客。中古で買ったハイラックスが今朝エンジンがかからない（case_001） |
| 顧客B | コートジボワールの顧客。並行輸入したハイラックスのナビ画面がタイ語のまま（case_005） |

## Before：現状の課題を最初に共有する（デモ導入・2分）

デモに入る前に、現状の対応フローを一枚で見せて課題認識を揃える。

> 顧客から問い合わせ → 担当者が記憶とマニュアルを探し回る（**平均5〜10分**）→ 分からなければベテランに聞く／保留にする → 回答 → **対応内容はどこにも構造化されて残らない** → 同じ質問に別の担当者がまた5〜10分かける → 解決できなかった質問は「なかったこと」になり、FAQもマニュアルも改善されない

ここで顧客に問いかける:「**解決できなかった問い合わせが、先月何件あったか分かりますか？**」——多くの場合答えられない。これが本PoCで解決したいことだと伝えて、デモに入る。

## シーン1：新人でも迷わず対応できる（case_001 / 約3分）

**狙い＝メリット①「情報を探す時間の削減」を体感させる**

### 進行
1. Aminaの立場で `/assist` を開く。受付情報を選択（Voice / Kenya / Toyota Hilux / Certified used / Vehicle operation）
2. 顧客Aの訴えをそのまま入力:
   > My Hilux won't start this morning. The starter clicks once but the engine doesn't turn over.
3. **Search with AI** をクリック。数秒で以下が表示される:
   - 回答候補2件（第一候補: バッテリー・端子腐食の可能性と対処手順／第二候補: スターターソレノイド等の別原因）
   - 確認質問「メーターの照明は暗くないか？バッテリーはいつ交換したか？」
   - 参考資料（オーナーズマニュアル該当章、ジャンプスタートFAQ）
   - 次アクション（端子確認とジャンプスタートの案内→ダメなら入庫予約）
4. 確認質問を顧客に聞き、回答をメモ。第一候補を採用（チェック）し、次アクションを実施チェック
5. 解決結果を **Resolved** で記録して保存

### ここで顧客に伝えること
- 「Aminaはマニュアルを1ページも開いていません。**探す時間がゼロ**です。従来5〜10分かかっていた調べ物が、入力と確認だけになります」
- 「確認質問と次アクションは、**ベテランなら必ず聞くこと・やること**です。つまり新人のAminaが、初日からベテランの型で対応できます」

### 要件ディスカッションポイント（その場でメモを取る）
- 実際のナレッジソースは何か（既存FAQ／整備マニュアル／過去対応履歴／部品システム？）
- 受付時に選択する項目はこれで足りるか（顧客ID連携、VIN入力は必要か）
- 音声通話の場合、質問入力は担当者の手入力でよいか、音声認識まで踏み込むか

## シーン2：「勝ちパターン」が組織に蓄積される（case_008 / 約2分）

**狙い＝メリット②「解決率を上げる勝ちパターンの蓄積」を体感させる**

### 進行
1. 続けてもう1件。タイヤ空気圧警告灯の問い合わせ（case_008）を入力:
   > The tyre pressure warning light stays on even after I inflated all four tyres.
2. AIがTPMSリセット手順・ドア内側の指定空気圧の確認・（再発時の）スローパンク点検という**過去に解決実績のある型**を提示
3. どの回答候補を採用したかをチェックして解決記録

### ここで顧客に伝えること
- 「大事なのは、Aminaが**どの回答で解決できたかが記録される**ことです。採用された回答は『勝ちパターン』としてスコアが上がり、次に同じ質問が来たとき、より確度の高い候補として先頭に出てきます」
- 「つまり、**対応すればするほど、組織全体の解決率が上がっていく**仕組みです。ベテランの暗黙知が、辞めても残る形になります」
- 補足として、給油所の空気圧計が不正確なことが多いという**現地特有のノウハウ**も候補に含まれていることを見せ、「地域の実情に合わせたナレッジが育つ」ことを印象付ける

### 要件ディスカッションポイント
- 「採用された回答」の重み付けロジックはPoCでどこまで作り込むか（単純な採用率か、解決率との掛け合わせか）
- 回答テンプレートの管理者承認フローは必要か

## シーン3：解決できなかった問い合わせが「資産」になる（case_005 / 約3分）

**狙い＝メリット③「未解決ログの蓄積・分析→ナレッジギャップ特定」を体感させる。本デモのクライマックス。**

### 進行
1. 顧客B（コートジボワール、フランス語）から難しい問い合わせが来る。受付を Text (WhatsApp) / Côte d'Ivoire / Hilux / **Grey import** に切り替えて入力:
   > The navigation and all menus are in Thai. How do I change the system to French?
2. AIの第一候補は「正規車なら設定変更可、ただしこの車はタイ仕様の並行輸入車のため、フランス語パックが入っていない可能性」と**確信度Medium**で提示。変換対応の可否・価格はナレッジベースに存在しないことが示唆される
3. 次アクションに「インフォテインメント専門担当へエスカレーション」「**ナレッジギャップとして記録**」が出る
4. 解決結果を **Escalated** で記録

### ここで顧客に伝えること
- 「従来なら、この対応は『答えられなかった、残念』で終わり、**記録も残らず、翌月また別の担当者が同じことで困ります**」
- 「このシステムでは、答えられなかったことが**構造化されたログとして残ります**。何を聞かれ、何を検索し、なぜ解決できなかったか、まで」

### そのままダッシュボードへ（Josephの視点に切り替え）
5. `/dashboard` を開く。今対応した3件が履歴とKPIに反映されていることを見せる
6. KPIカードの **Knowledge gaps** を指し示す:「エスカレした案件・AIの回答が採用されなかった案件が自動で集計されています」
7. 履歴テーブルで、並行輸入車関連のエスカレが繰り返し発生していることを見せる

### ここで顧客に伝えること（クロージング）
- 「Josephは月次会議で、勘ではなく**データで**『並行輸入車のインフォテインメント対応がナレッジの穴だ。変換メニューと価格表のFAQを作るべきだ』と言えるようになります」
- 「FAQが追加されれば、次から同じ質問はシーン1と同じように**その場で解決**できます。**今日の未解決が、来月の即答になる**——これがこのPoCで作りたい改善サイクルです」
- 冒頭の問いに戻る:「『解決できなかった問い合わせが先月何件あったか』に、**このダッシュボードは即答できます**」

### 要件ディスカッションポイント
- ナレッジギャップの定義はこれでよいか（エスカレ＋回答不採用。他に基準はあるか）
- FAQ改善の実施フロー（誰が書き、誰が承認し、どうAIに反映するか）はどの部門が担うか
- Phase 1で分析画面（/analysis）・改善提案画面（/suggestions）をどこまで作るか

## After：Before/Afterの対比でまとめる（2分）

| | Before | After（PoC後の姿） |
|---|---|---|
| 情報を探す時間 | 対応ごとに5〜10分 | 入力→数秒で候補提示（**削減**） |
| 対応品質 | 担当者の経験に依存 | 誰でもベテランの型で対応（**勝ちパターンの蓄積**） |
| 未解決の問い合わせ | 記録に残らず消える | ログとして蓄積・自動集計（**ナレッジギャップの特定**） |
| FAQ/マニュアル改善 | 気づいた人が思いつきで | データに基づく優先順位付け（**改善サイクル**） |

最後に次のステップを提示する:
1. 本日のディスカッションポイントを持ち帰り、**このモックをベースに要件定義**を開始する（対象ナレッジソース、対象国・チャネル、ギャップ定義、改善フロー）
2. PoC期間中に収集するログ項目とKPI（解決率、AI採用率、平均処理時間）の目標値を合意する
