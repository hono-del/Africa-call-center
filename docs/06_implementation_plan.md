# 06. 実装計画

> 本ドキュメントはまだコードに着手する前の計画書です。ここまでの `00`〜`05` の設計方針に基づき、既存モックからの変更方針・優先度・対象ファイル・実装順序を整理します。**対象ファイルパスはリポジトリ未参照のため推定です。実装開始前に実際のファイル構成と照合してください。**

## 1. 既存モックからの変更方針（サマリ）

| 既存モックの要素 | 変更方針 |
|---|---|
| `/call` 画面 | `/assist` に改称し、6ステップ（受付/入力/検索/提示/記録/保存）の構成に作り直す |
| `/dashboard` 画面 | KPIの土台は流用しつつ、チャネル別・言語別内訳を追加 |
| `/analysis` 画面 | 骨格は流用。カテゴリマスタ・車種マスタをアフリカ市場向けに刷新 |
| `/suggestions` 画面 | ほぼそのまま流用。改善アクションの種類（FAQ拡充/マニュアル改善/AI回答精度改善/トレーニング）を明確化 |
| `/captions` 画面 | 削除（Phase 1のスコープ外） |
| 車種・ブランドマスタ | 日本・北米寄りラインナップから、トヨタ（ハイラックス中心）・スズキ等アフリカ市場実態に合わせて総入れ替え |
| 問い合わせカテゴリ | メンテナンス/EV充電等から、部品供給・納期/悪路耐久性/並行輸入車対応等に総入れ替え |
| 対応言語 | 日/英/中/韓 → 英/仏/日 の3言語に変更 |
| 対応チャネル | 音声のみ想定 → 音声＋テキストチャネルに拡張 |
| データモデル | 検索ログ・回答候補・確認質問・次アクション・解決結果のフィールドを新規追加（`04_data_model.md`参照） |

## 2. 優先度A/B/C

### 優先度A（Phase 1のコア。これがないとデモの価値が伝わらない）
- `/assist` 画面の新規実装（6ステップ全体、特に「回答候補が複数」「確認質問」「次アクション」「解決可否の必須記録」）
- 新規データモデル（`InquiryLog` / `SearchLog` / `AnswerCandidate` / `Resolution`）の実装
- アフリカ市場向け車種・ブランド・カテゴリのモックデータ作成
- `/captions` 画面・関連コードの削除

### 優先度B（デモの説得力を高めるが、なくても中核体験は伝わる）
- `/dashboard` のチャネル別・言語別内訳の追加
- `/analysis` のカテゴリマスタ刷新、ナレッジギャップ検出ロジックの調整（`AnswerCandidate.wasAdopted`ベースへの変更）
- `/suggestions` の改善アクション種別の明確化、`ImprovementCandidate`データモデルへの対応
- 英語・フランス語・日本語のi18n切り替えの実装

### 優先度C（Phase 1完了後でも問題ない。将来フェーズ寄り）
- 販売店スタッフ・エンドユーザー向け画面の設計余地の作り込み（権限モデルの拡張等）
- 商材マスタの汎用化（`data/products/automotive.ts`のような分離。Phase 1では自動車のみのため優先度は低いが、後の手戻りを避けるため早めの分離が望ましい）
- 通信環境を考慮した軽量化・オフライン耐性
- デモ用シナリオデータ（`05_demo_scenario.md`のサンプルケース）を固定データとして用意する

## 4. 実装順序（推奨）

1. **データモデルの実装**（`04_data_model.md`の型定義をコード化）: `InquiryLog` `SearchLog` `AnswerCandidate` `Resolution` `ImprovementCandidate`
2. **アフリカ市場向けマスタデータの作成**: 車種・ブランド・カテゴリ（実際の市場シェア感を反映）
3. **`/assist` 画面の実装**（優先度A、Phase 1の核）
   - 受付→入力→検索→提示→記録→保存の6ステップを画面上に実装
   - 解決可否が未入力の場合は対応終了できないバリデーションを実装
4. **`/captions` 画面・関連コードの削除**
5. **`/dashboard` のKPI追加**（チャネル別・言語別内訳）
6. **`/analysis` のマスタ刷新・ギャップ検出ロジック調整**
7. **`/suggestions` の改善アクション種別対応**
8. **i18n（英/仏/日）の実装**
9. **デモ用シナリオデータの固定化**（`05_demo_scenario.md`の3分デモケースを再現できるようにする）
10. **将来拡張（商材マスタの汎用化、権限モデルの拡張）は余力があれば着手**

## 5. リスク・注意点

- `/assist` の再設計はUI・データモデルともに範囲が広いため、最初にモックデータ（固定のJSON等）で画面を作り切ってから、動的なロジックに拡張する進め方を推奨する
- 解決可否の必須入力バリデーションは、KPIの信頼性に直結するため、実装漏れがないよう`03_screen_design.md`のチェック項目と突き合わせる
- 既存の`/call`から`/assist`への改称に伴い、内部の変数名・ルーティング・ドキュメント上の呼称を統一する（旧名称の残存に注意）
