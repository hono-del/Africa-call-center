# 05. デモシナリオ

## 1. 社内説明用デモシナリオ（詳細版・所要目安10〜15分）

**目的**: 社内の意思決定者・関係部門に対して、PoCの狙いと今後の展開可能性を丁寧に説明する。

### 導入（1〜2分）
- 背景説明: アフリカ販売代理店のコールセンターでは、車種・言語・対応チャネルが多様であり、担当者の対応品質にばらつきが出やすいという課題認識を共有する
- 本PoCの目的を提示: 「AIナレッジ検索・回答支援」がどのように担当者を支援し、その結果がどう組織の改善サイクルにつながるかを見せる

### 本編（7〜10分）
1. **`/assist` 画面のデモ**（最重要パート）
   - 例: ケニアの顧客からハイラックスの部品供給についてWhatsAppで問い合わせが入った、という設定でデモを開始する
   - 症状・質問を入力 → AI検索を実行
   - 回答候補（複数）、確認質問、参考資料、次アクションが提示される様子を見せる
   - 確認質問に対する回答を記録し、回答候補を選択
   - 解決可否（この例では「解決済み」）を記録し、対応終了
2. **`/dashboard` 画面のデモ**
   - 今の対応が件数・カテゴリ別集計にどう反映されるかを見せる
   - チャネル別（音声/テキスト）・言語別（英/仏/日）の内訳を紹介する
3. **`/analysis` 画面のデモ**
   - 蓄積されたログから、特定カテゴリ（例: 並行輸入車のサポート）でナレッジギャップが多いことが可視化される様子を見せる
4. **`/suggestions` 画面のデモ**
   - そのギャップに対して「並行輸入車の保証範囲FAQを追加」という改善提案が自動生成されている様子を見せる
   - 改善提案が実行されればFAQが更新され、次回以降のAI回答精度が向上する、という循環構造を説明する

### まとめ（2〜3分）
- Phase 1の範囲（担当者・管理者向け）と、将来の拡張余地（販売店スタッフ・エンドユーザー、他商材への展開、実チャネル連携）を提示する
- 今後の検討事項（対象国の絞り込み、実データ・実API連携の要否等）を共有し、次のディスカッションにつなげる

## 2. 顧客説明用デモシナリオ（簡潔版・所要目安5〜7分）

**目的**: 顧客（代理店側の意思決定者）に対して、導入した場合の価値を端的に伝える。

### 流れ
1. 「コールセンターの対応品質を均一化し、蓄積したログが自動的に改善につながる仕組み」という一言で価値を提示する
2. `/assist` 画面のデモで、担当者が症状を入力するだけでAIが回答候補・確認質問・次アクションを提示する様子を見せ、「経験の浅い担当者でも一定水準の対応ができる」という価値を強調する
3. `/analysis` `/suggestions` 画面で、「対応した内容が自動的に分析され、FAQやマニュアルの改善点が見えてくる」という価値を強調する
4. 多言語対応（英語・フランス語）に触れ、複数国展開を見据えた設計であることを伝える
5. 質疑応答

## 3. 3分デモの流れ（エレベーターピッチ版）

**目的**: 短時間で核心の価値を伝える。細部の説明は省略し、体験の流れに絞る。

| 時間 | 内容 |
|---|---|
| 0:00-0:30 | 課題提示: 「アフリカの代理店コールセンターは車種・言語・チャネルが多様で、対応品質にばらつきが出やすい」 |
| 0:30-1:30 | `/assist` デモ: 症状入力→AI検索→回答候補・確認質問・次アクション提示→解決記録、という一連の流れを実際に操作して見せる |
| 1:30-2:15 | `/analysis` `/suggestions` デモ: 蓄積ログからナレッジギャップが可視化され、改善提案が自動生成される様子を見せる |
| 2:15-3:00 | まとめ: 「個々の対応が組織のナレッジ改善に自動的につながる」という価値を一言でまとめ、Phase 1以降の展開可能性（多言語・他商材・実チャネル連携）に触れて締める |

### 3分デモで使うサンプルケース（統一しておくと説明がぶれない）
- 顧客: コートジボワール（フランス語圏）の代理店経由の顧客
- 車両: トヨタ ハイラックス（並行輸入車）
- 問い合わせ内容: 部品供給の納期についての質問
- チャネル: WhatsApp（テキスト）
- 結果: 確認質問に回答後、回答候補を採用し「解決済み」として記録
